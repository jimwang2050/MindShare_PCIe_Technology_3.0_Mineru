## 多主机系统

如果存在多个根复合体 (Root Complex)(参见第 97 页的图 3‐6),则 Configuration Address 与 Configuration Data 端口可以在各自 Host/PCI 桥的相同 IO 地址处被复制。为避免冲突,只有一个桥会响应该处理器对配置端口的访问。

1. 当处理器启动对 Configuration Address Port 的 IO 写操作时,主桥被配置为只有一个桥会主动参与该事务。

2. 在枚举过程中,软件会发现并对活动桥之下的所有总线进行编号。完成之后,它会启用非活动的主桥,并为其分配一个位于已分配给活动桥范围之外的总线号,然后继续枚举过程。两个主桥都会看到这些 Request (请求),但由于它们的总线号不重叠,因此只响应对应总线号的请求,所以不会产生冲突。

3. 此后,对 Configuration Address Port 的访问会同时发往两个主桥,而后续对 Configuration Data Port 的读或写访问只会被作为目标总线网关的那个 Host/PCI 桥接受。该桥会响应该处理器的请求,另一个桥则忽略它。

o 如果目标总线是 Secondary Bus,该桥把访问转换为 Type 0 配置访问。

o 否则,将其转换为 Type 1 配置访问。

## 概述

在规范制定者选择 PCI‐X 以及后来的 PCIe 如何访问配置空间 (Configuration space) 时,面临两方面的考量。首先,每个 Function 仅 256 字节的空间限制了厂商在该空间内放置专有信息,也限制了未来的规范制定者——他们需要为更多标准化的 capability 结构预留空间。为解决该问题,空间直接从每个 Function 256 字节扩展为 4KB。其次,在 PCI 诞生之初,使用多处理器的系统寥寥无几。当系统中只有一颗 CPU 并且只运行一个线程时,旧模型需要两步才能完成一次访问,这并不构成问题。但较新的机器采用多核、多线程的 CPU,这对 IO 间接模型带来了问题,因为没有机制阻止多个线程同时访问配置空间。因此,如果不引入某种锁机制,这种两步模型将无法继续工作。在没有锁机制的情况下,一旦线程 A 将一个值写入

为了解决这个新问题，规范的编写者决定采用一种不同的方法。他们不再尝试节约地址空间，而是通过将全部配置空间映射到内存地址的方式，创建一个单步、不可中断的流程。由于在指定的地址范围内的一次内存请求将在总线上生成一次配置请求 (Configuration Request)，因此这使得整个过程只需要一条命令序列即可完成。如今的代价则在于地址大小。为每个 Function 映射 4KB 配置空间，并覆盖所有可能的实现方式，需要分配 256MB 的内存地址空间。就目前而言，现代架构通常支持 36 位到 48 位的物理内存地址空间，与之相比，256MB 已经微不足道。

## Configuration Requests (配置请求)

Two request types, Type 0 or Type 1, may be generated by bridges in response to a configuration access. The type used depends on whether the target Bus number matches the bridge's Secondary Bus Number, as described below.

针对一次配置访问,桥可以生成两种请求类型:Type 0 或 Type 1。所使用的类型取决于目标 Bus 号是否与桥的 Secondary Bus Number 相匹配,具体见下文描述。

---

## Type 0 Configuration Request

如果目标 Bus Number 与 Secondary Bus Number 匹配,则会将 Type 0 配置读或写请求转发到 secondary bus,并:

1. 该 Bus 上的各 Device 通过检查 Device Number 来确定哪一个是目标 Device。请注意,外部 Link 上的端点 (Endpoint) 始终是 Device 0。

2. 被选中的 Device 通过检查 Function Number 来确定在该 Device 内选中的是哪一个 Function。

3. 被选中的 Function 使用 Register Number 字段在其配置空间中选择目标 dword,并使用 First Dword Byte Enable 字段选择在该 dword 内要读取或写入的字节。

## Type 1 Configuration Request

当 Bridge 收到一个 Configuration 访问请求,其目标 Bus Number 与自身的 Secondary Bus Number 不一致,但落在 Secondary Bus Number 与 Subordinate Bus Number 的范围之间时,它会将该报文以 Type 1 Request 的形式转发到自身的 Secondary Bus 上。不是 Bridge 的设备(Endpoint)由于目标位于不同的 Bus 上,会忽略该 Type 1 Request;而 Bridge 收到该报文后,会对目标 Bus Number 与其下游 Bus 范围做同样的比较(参见第 87 页的图 3-1 和第 97 页的图 3-6)。

如果目标 Bus 与 Bridge 的 Secondary Bus 匹配,则该报文会从 Type 1 转换为 Type 0,并传递到 Secondary Bus 上。该 Bus 上的本地设备随后按前述方式检查报文 Header。

- 如果目标 Bus 不是 Bridge 的 Secondary Bus,但在其范围内,则该报文会以 Type 1 Request 的形式转发到 Bridge 的 Secondary Bus。

## PCI 兼容配置访问示例

参考第 104 页的图 3-9。为了说明使用传统 CF8h/CFCh 机制生成配置请求的概念,请参考以下 x86 汇编代码示例。该示例会使根复合体 (Root Complex) 从 Bus 4、Device 0、Function 0、Register 0(Vendor ID)执行一次 2 字节的读操作。

```csv
mov dx,0CF8h ;set dx = config address port address
mov eax,80040000h;enable=1, bus 4, dev 0, func 0, DW 0
out dx,eax ;IO write to set up address port
mov dx,0CFCh ; set dx = config data port address
in ax,dx ;2-byte read from config data port
```

1. `out` 指令从处理器生成一次 IO 写,目标是根复合体 (Root Complex) Host bridge 中的 Configuration Address Port(0CF8h),如第 92 页的图 3-4 所示。

2. Host bridge 将 Configuration Address Port 中指定的目标总线号(4)与其下游所辖的总线范围(0~10)进行比较。目标总线落在该范围内,因此该桥已被预置好下一次配置请求的目的地。

3. `in` 指令从处理器生成一次 IO 读事务,目标是根复合体 (Root Complex) Host bridge 中的 Configuration Data Port。这是从 Configuration Data Port 起始两个位置进行的 2 字节读取。

4. 由于目标总线并非 Bus 0,Host/PCI 桥在 Bus 0 上发起一次 Type 1 配置读。

5. Bus 0 上的所有设备都会锁存该事务请求,并识别出它是一个 Type 1 配置请求 (Configuration Request)。因此,根复合体 (Root Complex) 中的两个虚拟 PCI-to-PCI 桥都会将 Type 1 请求中的目标总线号与各自下游的总线范围进行比较。

6. 目标总线(4)位于左侧桥的下游总线范围之内,因此它将该报文透传到其 Secondary Bus,但仍以 Type 1 请求的形式转发,因为目标总线与 Secondary Bus Number 不匹配。

7. 左侧交换机的上游端口接收到该报文,并将其递交给上游的 PCI-to-PCI 桥。

8. 该桥确定目标总线位于其下游,但并非指向其 Secondary Bus,因此它以 Type 1 请求的形式将该报文转发到 Bus 2。

9. Bus 2 上的两个桥都会接收到该 Type 1 请求报文。右侧的桥确定目标总线与其 Secondary Bus Number 相匹配。

10. 该桥将配置读请求透传到 Bus 4,但由于报文已达到目标总线(目标总线号与 Secondary Bus Number 相匹配),因此将其转换为 Type 0 配置读 (Configuration Read) 请求。

11. Bus 4 上的 Device 0 接收该报文,并解码目标 Device、Function 与 Register Number 字段,以选择其配置空间中的目标双字(参见第 90 页的图 3-3)。

12. First Dword Byte Enable 字段中的 bit 0 和 bit 1 被置位,因此该 Function 在完成报文 (Completion) 中返回其前两个字节(本例中即 Vendor ID)。完成报文 (Completion) 使用从 Type 0 请求报文中获取的请求者 ID (Requester ID) 字段路由回 Host bridge。

13. 这两个字节的读数据被递交给处理器,从而完成 `in` 指令的执行。Vendor ID 被放入处理器的 AX 寄存器中。

## 发现 Function 的存在与缺失

处理器上执行的配置软件通常通过读取 Function 的 Vendor ID 寄存器来发现其是否存在。PCI-SIG 为每个供应商分配一个唯一的 16 位数值,并将该值硬连线到该供应商所设计的每个 Function 的 Vendor ID 寄存器中。通过在系统中所有可能的 Bus、Device 和 Function 编号组合下读取此寄存器,枚举软件可以遍历整个拓扑结构,从而获知哪些设备存在。这一过程相当简单,但可能出现两个问题:目标设备可能不存在,或者目标设备存在但尚未准备好响应。对这两种情况的处理将在下文描述。

---

## 设备不存在 (Device not Present)

在发现 (discovery) 过程中,目标设备实际上并不存在于系统中的情况会发生多次,这种情况发生时必须被正确理解。在 PCI 中,配置读请求 (Configuration Read Request) 会在总线 (bus) 上超时并产生 Master Abort 错误条件。由于没有设备驱动总线,且所有信号都被上拉,总线上的数据位将被视为全 1,并成为所看到的数据值。得到的 Vendor ID 为 FFFFh,该值已被保留。若枚举软件 (enumeration software) 看到读操作返回此结果,便知道该设备不存在。由于这并非真正的错误条件,Master Abort 不会在枚举过程中作为错误被报告。

对于 PCIe 而言,对不存在设备的配置读请求会导致目标设备上方的桥返回一个状态为 UR (Unsupported Request, 不支持的请求) 的无数据完成报文 (Completion)。为向后兼容传统枚举模型,根复合体 (Root Complex) 在枚举期间遇到此类完成报文时,会向处理器返回全 1 (FFFFh) 作为数据。请注意,枚举软件依赖于在探测系统中各功能 (Function) 是否存在时,收到无不支持请求 (Unsupported Request) 返回的配置读请求对应的全 1 值。

避免在此情况下误报错误非常重要。尽管这种超时或 UR 结果在运行时会被视为错误,但它在枚举期间属于预期结果,不被认为是错误。为帮助避免由此带来的混淆,设备通常不会被使能在稍后才报告错误。在 PCIe 中,记录这一事件仍可能有其价值,这便是 PCIe 能力寄存器 (Capability) 块中提供第四个 "错误" 状态位——即 Unsupported Request Status(不支持的请求状态)——的原因(更多内容请参阅第 678 页的 "使能/禁止错误报告")。这样可以在不将其标记为错误的前提下记录该条件,这一点很重要,因为检测到的错误可能会停止枚举过程而调用系统错误处理例程。此时错误处理软件的能力可能有限,从而难以解决问题。枚举软件在这种情况下可能失败,因为它通常是在操作系统或其他错误处理软件可用之前执行的。为避免此风险,在枚举期间通常不应报告错误。

## Device not Ready

另一个可能出现的问题是,目标设备已存在但尚未准备好响应当前配置访问。由于设备需要时间准备以接受访问,因此配置操作存在一个时序上的考量。当数据速率为 5.0 GT/s 或更低时,软件必须在复位 (Reset) 之后等待 100ms 才能发起配置请求 (Configuration Request)。当速率高于 5.0 GT/s(Gen3 (8.0 GT/s) 速率)时,软件必须等待到链路 (Link) 训练完成之后 100ms 才能尝试访问。对于高速情况需要更长延迟的原因是:在链路 (Link) 训练期间的 Gen3 (8.0 GT/s) 均衡 (Equalization) 流程可能会耗费较长的时间(大约 50ms;详见第 577 页"链路均衡概述"以获取更多相关信息)。

按照 PCI 2.3 规范的定义,初始化时间 $( \mathrm { { T _ { r h f a } } }$ ‐ 从复位 (Reset) 解除到首次访问的时间)从 RST# 撤销开始,经过 $2 ^ { 2 5 }$ 个 PCI 时钟后结束。这相当于整整一秒的时间,期间功能 (Function) 正在为首次配置访问做准备,这一数值在 PCIe 中沿用为 1.0s(+50%/‐0%)。功能 (Function) 可以利用这段时间来填充其配置 (Configuration) 寄存器,例如从外部串行 EEPROM 中加载内容。加载过程可能需要较长时间,在完成之前,功能 (Function) 将无法正确响应访问。在 PCI 中,如果在功能 (Function) 准备好之前收到配置访问,它有三种选择:忽略该请求 (Request)、重试该请求 (Request),或者接受该请求 (Request) 但推迟到完全就绪后再返回响应。最后一种响应方式会给热插拔 (Hot-Plug) 系统带来麻烦,因为共享总线可能会被阻塞长达一秒,直到该请求 (Request) 被处理完毕。

在 PCIe 中我们也面临同样的问题,但处理流程稍有不同。首先,当 PCIe 功能 (Function) 暂时无法响应当前配置访问时,必须始终返回带有特定状态的完成报文 (Completion),该状态即配置请求重试状态 (Configuration Request Retry Status,CRS)。此状态仅在响应当前配置请求时合法,若在响应其他请求 (Request) 时出现,可选择将其视为畸形报文 (Malformed Packet) 错误。该响应同样仅在复位 (Reset) 后的一秒内有效,因为功能 (Function) 应在此期间内给出响应,否则可视为该功能 (Function) 已损坏。

根复合体 (Root Complex) 处理针对配置读请求 (Configuration Read Request) 所返回的 CRS 完成报文 (Completion) 的方式是实现 (implementation) 特定的,但系统复位 (Reset) 后的那一段时间除外。在那一段时间内,根据其根控制寄存器 (Root Control Register) 中 CRS 软件可见性 (CRS Software Visibility) 位的设置(如图 3‐11 所示,见第 108 页),根复合体下一步有两种处理方式:

如果该位被置位,并且当前请求 (Request) 是对 Vendor ID 寄存器两个字节的配置读(这与枚举访问为探测功能 (Function) 是否存在时所做的访问一致),则根复合体 (Root Complex) 必须向主机返回该寄存器的伪造值 0001h,并将该请求 (Request) 中其他所有字节返回全 1。该 Vendor ID 不用于任何真实设备,软件会将其解读为访问该设备可能存在较长延迟的指示。这很有用,因为软件可以选择转去执行其他任务,从而更好地利用原本用于等待设备响应的时间,稍后再回过头来查询该设备。要使该机制生效,软件必须确保在复位 (Reset) 条件之后对功能 (Function) 的首次访问是对 Vendor ID 寄存器两个字节的配置读。

- 对于配置写操作或任何其他配置读操作,根复合体 (Root Complex) 必须自动以新请求的形式再次发出该配置请求 (Configuration Request)。

## 判断一个 Function 是 Endpoint 还是 Bridge

枚举过程中的一个关键环节是要能够判断一个 Function 是桥 (Bridge) 还是端点 (Endpoint)。如第 108 页的图 3-12 所示,Header Type 寄存器(配置空间 Header 中偏移地址 0Eh)的低 7 位标识该 Function 的基本类别,共定义了三个取值:

- 0 = 不是桥 (在 PCIe 中为 Endpoint)

- 1 = PCI‐to‐PCI 桥 (缩写为 P2P),用于连接两条总线

- 2 = CardBus 桥 (传统的接口,目前已较少使用)

在第 87 页的图 3-1 中,每个 Virtual P2P 的 Header Type 字段(DW3,byte 2)返回值应为 1,PCI Express‐to‐PCI 桥(Bus 8,Device 0)同样返回 1,而各个 Endpoint 则应返回 Header Type 为 0。

## 单根枚举示例

既然我们已经讨论了枚举过程中涉及的基本元素,接下来让我们走一遍该过程的一个示例。第 113 页的图 3‐13 展示了一个在总线和设备被枚举完成之后的系统示例。下面的讨论假定配置软件使用本章中定义的两种配置访问机制中的任意一种来达到此结果。系统启动时,在处理器上执行的配置软件会按下述步骤进行枚举。

1. 软件将 Host/PCI 桥的 Secondary Bus Number 更新为 0,将 Subordinate Bus Number 更新为 255。将其设置为最大值意味着在所有下游总线号被识别出来之前都不必再更改它。暂时将总线 0 到 255 都标识为下游。

2. 从 Device 0(桥 A)开始,枚举软件尝试读取 bus 0 上 32 个可能设备中 Function 0 的 Vendor ID。如果从 Bus 0、Device 0、Function 0 返回了有效的 Vendor ID,则说明该设备存在并且至少包含一个 Function。如果没有返回,则继续探测 bus 0、device 1、Function 0。

3. 本例中的 Header Type 字段(第 108 页的图 3‐12)包含值 1(01h),表明这是一个 PCI‐to‐PCI 桥。Header Type 寄存器中的 Multifunction 位(bit 7)为 0,表明 Function 0 是该桥中唯一的 Function。规范并不排除在该 Device 内实现多个 Function,并且这些 Function 中的每一个反过来又可以代表其他虚拟 PCI‐to‐PCI 桥,甚至是非桥 Function。

4. 软件既然已找到一个桥,便执行一系列配置写操作,按如下方式设置该桥的总线号寄存器:

- Primary Bus Number Register = 0
- Secondary Bus Number Register = 1
- Subordinate Bus Number Register = 255

该桥现在知道直接连接的下游总线号为 1(Secondary Bus Number = 1),并且其下游最大总线号为 255(Subordinate Bus Number = 255)。

5. 枚举软件必须执行深度优先搜索。在继续发现 bus 0 上的其他 Device/Function 之前,它必须先继续搜索 bus 1。

6. 软件读取 Bus 1、Device 0、Function 0 的 Vendor ID,在本例中即桥 C。返回了有效的 Vendor ID,表明 Bus 1 上存在 Device 0、Function 0。

7. Header 寄存器中的 Header Type 字段包含值 1(0000001b),表明这是另一个 PCI‐to‐PCI 桥。与之前一样,bit 7 为 0,表明桥 C 是一个单 Function 设备。

8. 软件现在执行一系列配置写操作,按如下方式设置桥 C 的总线号寄存器:

- Primary Bus Number Register = 1
- Secondary Bus Number Register = 2
- Subordinate Bus Number Register = 255

9. 继续进行深度优先搜索,从 bus 2、device 0、Function 0 的 Vendor ID 寄存器执行一次读操作。本例假设桥 D 是 Bus 2 上的 Device 0、Function 0。

10. 返回了有效的 Vendor ID,表明 bus 2、device 0、Function 0 存在。

11. Header 寄存器中的 Header Type 字段包含值 1(0000001b),表明这是一个 PCI‐to‐PCI 桥,并且 bit 7 为 0,表明桥 D 是一个单 Function 设备。

12. 软件现在执行一系列配置写操作,按如下方式设置桥 D 的总线号寄存器:

- Primary Bus Number Register = 2
- Secondary Bus Number Register = 3

15. Header 寄存器中的 Header Type 字段包含值 0(0000000b),表明这是一个 Endpoint Function。由于它是端点而非桥,它具有 Type 0 Header,并且其下不存在 PCI 兼容总线。此时,bit 7 为 1,表明这是一个多功能设备。

参考图 3-14(第 116 页)所示的多根系统。在该系统中,每个根复合体 (Root Complex):

## 多根枚举流程

在对图 3-14 (第 116 页) 中左侧树形结构进行枚举的过程中,次级根复合体 (Root Complex) 内的 Host/PCI 桥会忽略所有的配置访问,因为目标总线号不大于 9。需要注意的是,虽然 Bus 8 已被检测并编号,但其上没有连接任何设备。一旦该枚举过程完成,枚举软件将执行以下步骤来枚举次级根复合体:

1. 枚举软件将次级根复合体中 Host/PCI 桥的 Secondary Bus Number (次级总线号) 和 Subordinate Bus Number (下级总线号) 寄存器的值修改为总线 64 (本例中)。(在多根系统中,64 和 128 常被用作起始总线号,但这仅仅是软件上的惯例。PCI 或 PCIe 规范中并没有任何规则强制要求这种配置。在本例中,完全可以从总线 10 开始为次级根复合体分配总线号。)

2. 随后,枚举软件从总线 64 开始扫描,发现了连接到下游根端口 (Root Port) 的桥。

3. 执行一系列配置写操作,将其总线号寄存器设置如下:

• Primary Bus Number Register (主总线号寄存器) = 64

• Secondary Bus Number Register (次级总线号寄存器) = 65

• Subordinate Bus Number Register (下级总线号寄存器) = 255

此时,该桥能够识别出:其下游侧直连的总线号为 65 (Secondary Bus Number = 65),以及其最远下游侧的总线号同样为 65 (Subordinate Bus Number = 65)。

4. 在总线 65 上发现 Device 0,它仅实现了 Function 0;进一步扫描表明总线 65 上没有其他设备,因此搜索过程回退到上一级总线。

5. 枚举在总线 64 上继续,但未发现其他设备,因此 Host/PCI 桥的 Subordinate Bus Number (下级总线号) 被更新为 65。

6. 至此,枚举过程完成。

## 热插拔 (Hot-Plug) 相关考量

在热插拔 (Hot-Plug) 环境中,即允许在系统运行时添加或移除插卡的环境中,第 116 页图 3‐14 中 Bus 8 所呈现的情形可能会带来麻烦。当系统已完成枚举并处于正常运行状态时,如果将一块带有桥片的卡插入 Bus 8,就可能出现问题。该桥片的 Secondary Bus Number 和 Subordinate Bus Number 必须分配为高于其 Primary Bus 总线号,且范围必须完全包含其下游所需的全部总线号。原因是这些总线号必须落在新插入卡的上游桥片 Secondary Bus Number 和 Subordinate Bus Number 所界定的范围之内。

一种可行的做法是:为位于 Bus 8 上的桥片分配所需的 Bus Number,并将当前的 Bus 9 递增为比之前总线号大 1 的值,从而为新增的总线腾出空间。在系统运行时对总线号进行重排虽然可以做到,但有经验的人表示这很难做得很完善。

针对这一潜在问题,还存在一种更简单的解决方案:每当发现一个未占用的槽位时,只需保留一个总线号间隙即可。例如,当 Bus 8 已分配,而其下方存在一个空闲槽位时,可将下一个被发现的总线分配为更高的号码(例如 19 而非 9),以便为这些可热插拔场景预留足够的资源。这样,如果再插入一块带有桥片的卡,新的总线号就可以分配为 Bus 9,而不致引发任何麻烦。在大多数情况下,保留一个总线号间隙不会带来问题,因为系统总共可以分配多达 256 个总线号。

## PCI Express Technology

设备本身。这意味着这些内部位置需要可寻址。软件必须能够使用某个地址执行读或写操作,以访问目标设备内相应的内部位置。为实现这一点,这些内部位置需要从系统所支持的某个地址空间中分配地址。

PCI Express 支持与 PCI 完全相同的三个地址空间:

- Configuration (配置)

- Memory (内存)

- IO (IO)

## Configuration Space(配置空间)

正如我们在第 1 章中所看到的,Configuration Space(配置空间)是随着 PCI 一起引入的,目的是让软件能够以标准化的方式控制设备并查询其状态。PCI Express 在设计上与 PCI 保持软件向后兼容,因此配置空间仍然被支持并用于与 PCI 相同的目的。关于配置空间的更多信息(其用途、访问方式、大小、内容等)可在第 3 章找到。

尽管配置空间最初是为了保存标准化结构(PCI 定义的头标、能力结构等)而设计的,但在 PCIe 设备中,将设备特定的寄存器映射到其配置空间中是非常常见的做法。在这些情况下,映射到配置空间中的设备特定寄存器通常是控制、状态或指针寄存器,而非数据存储位置。

## 存储器地址空间与 IO 地址空间

## 概述

在 PC 发展的早期,IO 设备内部的寄存器/存储通过 IO 地址空间(由 Intel 定义)进行访问。然而,由于 IO 地址空间存在若干局限性和不良影响(此处不展开讨论),该地址空间很快就不再受软硬件供应商青睐。这导致 IO 设备内部的寄存器/存储被映射到内存地址空间中(通常称为内存映射 IO,即 MMIO)。然而,由于早期软件是针对使用 IO 地址空间访问 IO 设备内部寄存器/存储而编写的,因此业界普遍的做法是将同一组设备特定寄存器同时映射到内存
